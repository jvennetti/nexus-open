The Challenge
Real-World Scenario

This is the capstone. Every module led here. You're not building one tool or one automation — you're building a coordinated multi-agent system where four agents work from the same input and produce different outputs for different audiences.

The scenario is real. Organizations like this exist everywhere. The system you build here is the same architecture that appears inside real estate agencies, consulting firms, franchise operations, and any business where information comes in from multiple sources and needs to go out to multiple audiences in multiple formats.

You're building it in a controlled environment — all outputs land in your own documents and your own inbox. No external sends. No live systems. Real enough to learn from.

Step 1 — Generate your environment

Open Claude and paste this prompt to generate five documents for a non-profit called Meridian Missions:

"I'm building a multi-agent system for a non-profit called Meridian Missions. Generate the following five documents. Format each as a clearly labeled, separate section I can save individually. Document 1 — Org Brief: Meridian Missions' mission statement, the three regions they operate in (South Asia, East Africa, South America), their three donor segments (Standard, Partner, Major Donor), their communication tone, and what they care most about in how they present their work publicly. Document 2 — Six Field Reports: Two reports from each region (six total). Each report should include: team name, region, a 2–3 paragraph program update, two or three prayer requests, and exactly one item explicitly flagged as urgent. Make the urgent items varied — a medical situation, a funding gap, a security concern. Document 3 — Donor Contact List: 12 fictional donors. Four per segment (Standard, Partner, Major). Each entry: full name, a fictional email address, giving level, and which region they're most personally connected to. Document 4 — Donor Email Samples: One example donor update email for each segment. Standard gets a warm general update. Partner gets program specifics and impact numbers. Major Donor gets a personal note written as if from the executive director. Document 5 — Prayer Bulletin Sample: A complete example weekly prayer bulletin — what it includes, how it's structured, and how it reads when it's done well."

When Claude delivers the five documents: save each one, paste Document 3 into a Google Sheet with columns Name / Email / Giving Level / Region / Draft Email, read through all six field reports to confirm each has a clearly identifiable urgent item, and read the email samples and prayer bulletin — these are your quality bar. Every agent you build must produce output that clears this standard.

Your Challenge
  1. A1
    Agent 1 — The Field Summarizer. Start in Claude chat. Write a prompt that processes a single field report and extracts three items per report: a concise program progress summary, the prayer requests, and the urgent need — in a consistent format. Test against two or three reports until the output is clean and reliable. Bring this into Cowork as a Skill. Run it against all six reports. The output is a single document — six structured summaries, one per team — that serves as shared input for Agents 2, 3, and 4. Do not move on until every summary is accurate to its source report.
  2. A2
    Agent 2 — The Donor Communicator. Using the summaries from Agent 1, generate personalized donor update emails matched to each donor's giving level and regional connection. Standard donors receive the general program update for their region. Partner donors receive program detail and impact specifics. Major Donors receive a message written as if directly from the executive director. Write all output emails into the Draft Email column of your donor Google Sheet. Read every row — would a real donor recognize the difference between their segment and another's? If not, refine.
  3. A3
    Agent 3 — The Prayer Bulletin. Pull all prayer requests from Agent 1's six summaries and compile them into a single weekly bulletin — organized by region, formatted for email distribution, clean enough to send without editing. Compare the output against your Document 5 sample. Adjust until it matches the structure and tone. Save the result as a Google Doc titled: Meridian Missions — Prayer Bulletin [Week of (today's date)].
  4. A4
    Agent 4 — The Needs Router. Scan Agent 1's summaries for all urgent flags. For each of the three urgent items: draft a short priority alert message to the appropriate regional director, written as if from a staff member. Wire this agent to write to a new Google Sheet (Urgent Needs Tracker) with columns: Team Name, Region, Date Flagged, Urgent Need Description, Alert Drafted. Then send the three priority alert drafts to your own email address — one per region, with the regional director's name in the subject line. This is the agent that tests whether your system can take action, not just produce text.
  5. 5
    Run a full cycle. With all four agents built, run the system against your six field reports as if it's Monday morning and they just came in. Run in sequence: Agent 1 → six summaries → Agent 2 → twelve donor email drafts → Agent 3 → prayer bulletin → Agent 4 → three alert emails in your inbox. Produce every output with as little manual work as possible. Your deliverable is the real output of that full cycle.
Your Deliverable
Six structured summaries, twelve donor email drafts in the Google Sheet, a complete prayer bulletin saved to a Google Doc, a populated Urgent Needs Tracker, and three alert emails in your own inbox — produced in a single run. Every output should pass your quality bar before you mark this complete.
What you just built: The four-agent structure — ingest, communicate, distribute, escalate — is not specific to non-profits. It is the same pattern inside a real estate agency processing weekly market reports, a consulting firm managing client updates, or a franchise operation running location check-ins. You didn't just learn multi-agent architecture in theory. You ran it.
